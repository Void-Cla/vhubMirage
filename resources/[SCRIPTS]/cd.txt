# Diagnóstico Principal

O vHub está sofrendo de um fenômeno clássico:

**"CORE Frozen no papel, CORE ainda sendo usado como camada de negócio."**

A documentação diz:

* CORE = Kernel
* CORE = Estado
* CORE = Persistência
* CORE = Autoridade

Mas na prática:

* Garage toma decisões de persistência
* VehControl interfere no ciclo de vida do veículo
* SpawnSelector interfere em identidade/spawn
* Conce interfere em ownership
* Scripts externos assumem responsabilidades do CORE

Resultado:

Você tem uma arquitetura que tenta ser DDD/Ownership Driven, mas está operando como um monólito distribuído.

# Gap 1 — Vehicle Lifecycle Incompleto

Hoje não existe um fluxo único claramente definido para:

```text
COMPRA
↓
REGISTRO
↓
SPAWN
↓
ENTRADA
↓
USO
↓
SALVAMENTO
↓
GUARDAR
↓
DESPAWN
↓
PÁTIO
↓
RECUPERAÇÃO
```

Pelo que encontrei:

`vhub_vehcontrol`

envia:

```lua
Vehicle:onEnter(...)
```

para ativar a persistência.

Isso é um sinal vermelho.

O CORE deveria descobrir isso sozinho.

Se o `vEnter` falhar:

```text
fuel
damage
odometer
last_pos
engine
```

não acumulam.

Foi exatamente o bug citado no contexto.

# Gap 3 — Dupla Fonte de Verdade

Encontrei sinais disso.

Exemplo:

CORE salva estado.

VehControl atualiza estado.

Isso é perigoso.

A regra deveria ser:

```text
CORE
=
ÚNICO ESCRITOR
```

Scripts externos:

```text
request()
```

CORE:

```text
commit()
```

# Gap 4 — Spawn Ownership

Você mencionou:

> spawnselector não consegue controlar o ped

Isso normalmente indica:

### Cenário A

Algum recurso chama:

```lua
SetPlayerModel
```

antes do fluxo do SpawnSelector.

---

### Cenário B

Existe mais de um dono do spawn.

Exemplo:

```text
vhub_player_state
vhub_spawnselector
baseevents
spawnmanager
```

tentando spawnar simultaneamente.

---

### Cenário C

Ped recriado pelo CORE após SpawnSelector finalizar.

Muito comum.
